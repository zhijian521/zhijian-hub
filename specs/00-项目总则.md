# 00 · 项目总则

## 目标

本项目是基于 Next.js 的前端项目，致力于建立清晰的项目结构、稳定的架构边界和可执行的开发规范。

核心目标只有三个：

- 简洁：能复用就不重写，能删掉就不保留，能一层解决就不绕两层。
- 整齐：同类东西放在同类位置，命名、分层、依赖方向稳定一致。
- 完整：功能闭环，接口有文档，组件可预览，规则有校验兜底。

## 技术栈

| 项 | 选型 | 备注 |
|---|---|---|
| 框架 | Next.js 15 App Router | 推荐使用 App Router |
| 语言 | TypeScript 5 | 严格模式 |
| 运行时 | Node.js | API Route 运行在服务端 |
| 样式 | CSS Modules + CSS 变量 | 灵活选择 Tailwind/纯 CSS |
| 状态管理 | 按项目需要选择 | React Context / Zustand 等 |

新增依赖前，先按这个顺序自问：

1. 现有实现能不能复用？
2. Node/浏览器原生能力能不能解决？
3. 已安装依赖能不能解决？
4. 如果都不行，再引新依赖，而且必须能说明收益。

## YAGNI 优先级

写任何东西前，按这个顺序判断，停在第一个成立的位置：

1. 这东西现在真的需要吗？
2. 项目里已经有了吗？
3. 标准库能做吗？
4. 平台原生能做吗？
5. 已装依赖能做吗？
6. 能不能用最小实现做完？

不要为了“以后可能用到”提前建接口、目录、抽象层、工厂或空文档。

## 项目边界

- 项目功能模块化组织，公共能力按统一目录规范沉淀。
- 调整实现时优先保持产品行为稳定，不为了“更优雅”随意改动既有交互、接口语义或数据规则。
- 目录结构、模块边界、规范体系和代码组织方式必须长期稳定，避免随功能推进持续漂移。

## 明确不做

- 不做过于复杂的自动化工具链。
- 不做全局的空壳抽象或未使用的架构层。
- 不做只有一个实现的接口抽象。
- 不做没有实际用处的样板代码。

- 不做没有落地校验的“建议型规范”。

## 强制校验

提交前统一运行：

```bash
npm run check
```

当前 `check` 包含：

- `npm run lint`：ESLint 代码质量检查
- `npm run typecheck`：TypeScript 类型检查

如果只是大范围整理格式，可额外运行：

```bash
npm run format
```

但“这次改动是否完成”的判断，以 `npm run check` 为准。
