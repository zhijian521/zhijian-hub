# 07 · 测试规范

## 测试类型

### 单元测试 (Unit Test)

- **范围**：测试单个函数、组件或类的行为
- **特点**：速度快、隔离性好、依赖少
- **工具**：Vitest + React Testing Library
- **覆盖率目标**：核心业务逻辑 80%+

### 集成测试 (Integration Test)

- **范围**：测试多个组件或模块的协作
- **特点**：验证接口契约、数据流转
- **工具**：React Testing Library
- **覆盖率目标**：关键业务流 60%+

### 端到端测试 (E2E Test)

- **范围**：完整用户业务流程
- **特点**：真实浏览器环境、用户视角
- **工具**：Playwright / Cypress
- **覆盖率目标**：核心业务路径覆盖

## 单元测试规范

### 测试文件命名

```
src/utils/date.ts          # 源代码
src/utils/date.test.ts     # 对应的测试文件

src/components/button.tsx  # 组件代码
src/components/button.test.tsx  # 组件测试
```

### 测试结构

```ts
import { describe, it, expect, beforeEach } from 'vitest';

/*============================================================================
  date-utils — 日期处理工具函数测试

  测试日期格式化、比较、计算等核心功能
============================================================================*/

describe('date utils', () => {
  describe('formatDate', () => {
    it('should format date correctly', () => {
      // Arrange
      const date = new Date('2024-01-01');
      
      // Act
      const result = formatDate(date);
      
      // Assert
      expect(result).toBe('2024-01-01');
    });

    it('should handle null input', () => {
      // Act & Assert
      expect(() => formatDate(null)).toThrow('Invalid date');
    });
  });
});
```

### 测试命名规范

```ts
// ✅ 推荐：描述性名称
it('should return formatted date when valid input provided')
it('should throw error when date is null')
it('should handle timezone correctly')

// ❌ 不推荐：模糊不清
it('works correctly')
it('test case 1')
it('should work')
```

### AAA 模式

- **Arrange**：准备测试数据和环境
- **Act**：执行被测代码  
- **Assert**：验证执行结果

## 组件测试规范

### React 组件测试

```tsx
import { render, screen, fireEvent } from '@testing-library/react';
import { Button } from './Button';

describe('Button', () => {
  /*== 基础功能测试 ==*/
  it('should render button with text', () => {
    // Arrange
    render(<Button>Click me</Button>);
    
    // Act
    const button = screen.getByText('Click me');
    
    // Assert
    expect(button).toBeInTheDocument();
  });

  /*== 交互功能测试 ==*/
  it('should call onClick when button is clicked', () => {
    // Arrange
    const handleClick = vi.fn();
    render(<Button onClick={handleClick}>Click me</Button>);
    const button = screen.getByText('Click me');
    
    // Act
    fireEvent.click(button);
    
    // Assert
    expect(handleClick).toHaveBeenCalledTimes(1);
  });

  /*== 状态测试 ==*/
  it('should be disabled when disabled prop is true', () => {
    // Arrange
    render(<Button disabled>Disabled</Button>);
    
    // Act & Assert
    expect(screen.getByRole('button')).toBeDisabled();
  });
});
```

### 测试最佳实践

1. **测试用户可见行为**
   - 优先通过 `getByRole`、`getByText` 查询
   - 避免依赖实现细节

2. **避免测试实现细节**
   - 不测试内部状态
   - 不测试私有方法
   - 关注组件的公开接口

3. **使用合适的选择器**
   ```ts
   // ✅ 推荐
   screen.getByRole('button', { name: 'Submit' })
   screen.getByText('Welcome')
   screen.getByLabelText('Email')
   
   // ❌ 不推荐  
   screen.getByTestId('submit-btn')
   screen.getByClassName('button-primary')
   ```

4. **异步操作处理**
   ```tsx
   it('should display data after loading', async () => {
     render(<UserList />);
     
     // 等待异步数据加载
     await waitFor(() => {
       expect(screen.getByText('John Doe')).toBeInTheDocument(); 
     });
   });
   ```

## API 测试规范

### 接口测试

```ts
import { describe, it, expect, vi } from 'vitest';
import { userService } from './userService';

describe('userService', () => {
  beforeEach(() => {
    // 清理 mock
    vi.clearAllMocks();
  });

  /*== 成功情况测试 ==*/
  it('should fetch user list successfully', async () => {
    // Arrange
    const mockUsers = [{ id: 1, name: 'John' }];
    global.fetch = vi.fn().mockResolvedValue({
      ok: true,
      json: () => Promise.resolve(mockUsers),
    });

    // Act
    const users = await userService.getUsers();

    // Assert
    expect(users).toEqual(mockUsers);
    expect(global.fetch).toHaveBeenCalledWith('/api/users');
  });

  /*== 错误情况测试 ==*/
  it('should handle network error', async () => {
    // Arrange
    global.fetch = vi.fn().mockRejectedValue(new Error('Network error'));

    // Act & Assert
    await expect(userService.getUsers()).rejects.toThrow('Network error');
  });
});
```

## E2E 测试规范

### Playwright 测试示例

```ts
import { test, expect } from '@playwright/test';

test.describe('User Login', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/login');
  });

  test('should login successfully', async ({ page }) => {
    // Arrange - 填写登录表单
    await page.fill('[data-testid="email"]', 'test@example.com');
    await page.fill('[data-testid="password"]', 'password123');

    // Act - 点击登录按钮
    await page.click('[data-testid="login-btn"]');

    // Assert - 验证登录成功
    await expect(page).toHaveURL('/dashboard');
    await expect(page.locator('[data-testid="welcome-message"]')).toBeVisible();
  });

  test('should show error for invalid credentials', async ({ page }) => {
    // Arrange & Act
    await page.fill('[data-testid="email"]', 'wrong@example.com');
    await page.fill('[data-testid="password"]', 'wrongpassword');
    await page.click('[data-testid="login-btn"]');

    // Assert
    await expect(page.locator('[data-testid="error-message"]')).toContainText('Invalid credentials');
  });
});
```

## 测试覆盖率要求

| 类型 | 目标覆盖率 | 强制要求 |
|------|-----------|----------|
| 业务逻辑 | 80% | ✅ 必须 |
| API 服务 | 75% | ✅ 必须 |
| UI 组件 | 60% | ⚠️ 建议 |
| 工具函数 | 90% | ✅ 必须 |

## Testing Library 查询优先级

按照以下优先级使用查询方法：

1. **getByRole** - 首选，语义化最好
2. **getByLabelText** - 表单元素
3. **getByText** - 按钮、链接文本
4. **getByDisplayValue** - 输入框当前值
5. **getByAltText** - 图片 alt 文本
6. **getByTitle** - title 属性
7. **getByTestId** - 最后选择

## 禁止事项

- ❌ 不要测试实现细节
- ❌ 不要过度测试（测试覆盖率超过 100%没有意义）
- ❌ 不要用测试数据污染生产环境
- ❌ 不要把测试当作文档
- ❌ 不要在测试中包含敏感信息

## 测试最佳实践

1. **每个文件有对应测试**：重要模块必须有测试
2. **测试要快**：快速反馈，不阻塞开发
3. **测试要可靠**：不产生误报或漏报
4. **测试要简单**：易于理解和维护
5. **测试要及时**：在代码变更前编写测试
