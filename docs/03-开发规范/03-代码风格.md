# 03 · 代码风格

- 文档状态：开发规范
- 适用版本：`zhijian-hub@0.1.0`
- 最后更新：2026-07-18
- 维护范围：TypeScript、React 和通用代码风格
- 关联代码：`.editorconfig`、`.prettierrc`、`.eslintrc.json`、`tsconfig.json`

## 总则

代码读起来要像同一个人写的。优先追求：

- 简洁
- 稳定
- 易读
- 好改

注释解释“为什么这样做”，而不是逐行翻译“这行在做什么”。

## 文件头与分区注释

沿用项目里已经在使用的分区注释风格：

```ts
/*============================================================================
  api-response — 统一 API 响应结构

  所有 API Route 返回：
  成功 { code: 0, data, message }
  失败 { code: xxx, data: null, message }
============================================================================*/

/*== 业务错误码 ==*/
export const BizCode = { ... } as const;
```

三种注释形态，各有用途，不混用：

| 形态               | 用途                              | 示例                 |
| ------------------ | --------------------------------- | -------------------- |
| `/*== xxx ==*/`    | 分区标题，划开文件逻辑段          | `/*== 类型定义 ==*/` |
| `/*-- xxx --*/`    | 行内说明，紧跟被注释项            | props 字段、单行解释 |
| `/** ... */` JSDoc | 仅留给需要 IDE 悬浮提示的公共 API | `domain` 层导出函数  |

规则：

- 文件头用一段总注释说明"这个文件负责什么"和"关键约束是什么"。
- 文件超过约 40 行后，按逻辑分区，用 `/*== xxx ==*/` 划开。
- props 字段注释用 `/*-- xxx --*/` 行内形态，不写 JSDoc（除非是给外部消费的库 API）。
- 不要为了凑格式给很短、很直白的文件硬塞注释。

## import 分组与排序

每组之间空一行，组内按"第三方 → 项目别名 → 同目录相对"排：

```ts
// 1. Node 内置 / Next / 第三方
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

// 2. 项目内别名
import { BizCode, fail } from '@/lib/core/api-response';
import { requireAdminFromRequest } from '@/lib/core/auth';

// 3. 同目录相对路径
import { listSomething } from './queries';
```

每组前面加 `/*== xxx ==*/` 分区注释，统一用：

- `/*== 组件导入 ==*/` — React/Next 组件
- `/*== 数据与配置 ==*/` — `@/lib/*`、配置常量
- `/*== 样式导入 ==*/` — CSS Module
- `/*== 类型定义 ==*/` — interface/type 声明区

要求：

- 类型导入使用 `import type`。
- 跨目录统一使用 `@/*`，不要写层层相对路径。
- 只有一两行 import 的短文件可以省略分区注释。

## 命名

| 对象              | 规则                          | 示例                                |
| ----------------- | ----------------------------- | ----------------------------------- |
| 文件 / 目录       | kebab-case                    | `api-response.ts` `ghost-button/`   |
| 组件文件          | kebab-case；导出名 PascalCase | `ghost-button.tsx` -> `GhostButton` |
| 函数 / 变量       | camelCase                     | `requireAdminFromRequest`           |
| 类型 / 接口       | PascalCase                    | `SessionPayload`                    |
| 常量对象          | PascalCase                    | `BizCode`                           |
| CSS 变量          | `--kebab-case`                | `--space-4`                         |
| CSS Modules class | camelCase                     | `.submitButton`                     |

## TypeScript

- 开启严格模式，公共函数显式标注返回类型。
- 默认禁止 `any`。
- 只有在信任边界解析外部输入时，才允许短暂使用 `unknown` 或 `any`，并写注释说明原因。
- 类型尽量就近定义，不单独建一个"大杂烩 types 文件"。

## 工具链

| 命令                   | 用途                |
| ---------------------- | ------------------- |
| `npm run format`       | Prettier 直接格式化 |
| `npm run format:check` | 检查格式是否一致    |
| `npm run lint`         | ESLint 校验         |
| `npm run typecheck`    | TypeScript 校验     |
| `npm run test`         | Vitest 自动化测试   |
| `npm run check`        | 统一完成前校验入口  |

默认约定：

- 4 空格缩进
- 单引号
- 分号
- 尾逗号
- 长行交给 Prettier 处理，不手工对齐花样排版

## 简洁要求

- 先看有没有现成实现，再决定要不要新写。
- 同样的 401/403/错误响应出现两次以上，就该抽包装器或 helper。
- 能删的代码优先删，不要靠注释保留死代码。
- 函数尽量短，职责单一；如果滚过一整屏还没结束，基本该拆。

## 最小自检

非平凡逻辑需要留下最小可执行自检，但要和当前仓库能力匹配：

- 优先补强已有脚本或校验链，比如 `typecheck`。
- 纯函数或小算法可以留一个最小 `demo()` / 示例调用。
- 避免创建孤立未测试的代码文件。

也就是说，自检必须"可运行、可复现、可交给别人继续用"，而不是只在文档里说"以后再补测试"。

## 禁止项

- 禁止提交调试用 `console.log`。
- 禁止没有上下文的 `TODO`；如果必须留，写成 `TODO(name): reason`。
- 禁止空目录、空接口、空实现、空壳抽象。
- 禁止嵌套三元运算符（`a ? b : c ? d : e`）；多分支用 `if/else`、`switch` 或拆函数，单一条件的简短三元可以。
