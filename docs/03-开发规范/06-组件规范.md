## 组件分层规范

### 三层组件架构

| 层       | 目录                   | 定位                     | 依赖                     |
| -------- | ---------------------- | ------------------------ | ------------------------ |
| 原子组件 | `components/ui/`       | 无业务语义、可跨功能复用 | 通用工具                 |
| 站点组件 | `components/site/`     | 品牌和全站级展示元素     | UI 组件 + 站点样式       |
| 业务组件 | `components/features/` | 由多个组件组成的业务区块 | UI + 站点组件 + 业务逻辑 |

### 判断标准

- **原子组件**：Button、Input、Dialog 等基础 UI 元素，不包含业务逻辑
- **站点组件**：Brand、Header、Footer 等带有项目品牌或全站语义的组件
- **业务组件**：RoomCanvas 等绑定具体业务功能的组件

## 文件组织规范

### 目录结构

简单组件默认平铺，组件实现与同名 CSS Module 放在同一目录：

```text
components/ui/
├── button.tsx
└── button.module.css
```

当一个业务组件出现三个以上关联文件、多个内部子组件或独立测试资源时，再建立同名目录集中管理，例如 `components/features/room-canvas/`。不为简单组件额外创建 `index.ts` 和目录层级。

### 命名规则

- **目录名**: kebab-case（如 `ghost-button`）
- **文件名**: kebab-case（如 `ghost-button.tsx`）
- **导出名**: PascalCase（如 `GhostButton`）
- **样式**: CSS Modules + camelCase

## 组件实现规范

### 函数组件

```tsx
/*============================================================================
  ghost-button — 透明背景按钮组件

  提供不同主题的按钮变体，支持点击事件处理
============================================================================*/

/*== 依赖导入 ==*/
import React from 'react';

/*== 组件导入 ==*/

/*== 样式导入 ==*/
import styles from './button.module.css';

/*== 类型定义 ==*/
interface GhostButtonProps {
    /*-- 按钮内容 --*/
    children: React.ReactNode;
    /*-- 按钮主题，默认 default --*/
    variant?: 'default' | 'primary';
    /*-- 是否禁用，默认 false --*/
    disabled?: boolean;
}

/*== GhostButton 组件 — 透明背景按钮 ==*/
export function GhostButton({ children, variant = 'default', disabled = false }: GhostButtonProps) {
    return (
        <button className={styles.button} data-variant={variant} disabled={disabled}>
            {children}
        </button>
    );
}
```

### 最佳实践

1. **统一函数组件**：普通组件优先使用 `function`，`forwardRef` 等 React API 可使用具名函数表达式
2. **Props 类型**：使用 `interface`，就近定义
3. **默认值**：函数参数解构时设置
4. **样式优先**：静态视觉样式使用 CSS Modules；运行时坐标、尺寸和 CSS 变量可以使用内联 `style`
5. **副作用控制**：业务逻辑放在 hook 或容器组件
6. **注释规范**：文件头注释 + 分区注释 + 行内注释

## 优化清单

### 1. 文件结构与注释

- [ ] 文件头包含用途说明和关键约束
- [ ] 导入按依赖层级分组并注释
- [ ] Props 类型定义清晰，每个字段有说明
- [ ] 组件函数上方有功能描述注释
- [ ] JSX 关键区块有注释说明

### 2. 条件渲染优化

只有“满足条件时显示，否则不渲染”的单分支场景可以使用 `<Show>` 提升可读性：

```tsx
// 简单单分支可以直接使用 &&
{
    condition && <Component />;
}

// 条件较长或同文件需要统一表达时使用 Show
<Show when={condition}>
    <Component />
</Show>;
```

存在明确的两种展示结果时，可以使用简短三元表达式；多分支逻辑优先提前计算或拆成具名函数。不要为了统一形式增加无收益的包装层。

### 3. CSS 优化

- [ ] 样式按区块分组，用 `/*-- xxx --*/` 注释
- [ ] 颜色使用 CSS 变量（如 `var(--primary)`）
- [ ] 间距使用间距变量（如 `var(--space-4)`）
- [ ] 字号使用字号变量（如 `var(--text-base)`）
- [ ] 圆角使用圆角变量（如 `var(--radius)`）

### 4. 性能优化

- [ ] 先确认是否存在可感知或可测量的性能问题
- [ ] 昂贵计算或确有引用稳定需求时再使用 `useMemo` / `useCallback`
- [ ] 纯展示子组件频繁无效重渲染时再评估 `React.memo`
- [ ] 不为简单值、轻量计算和只在当前组件使用的回调提前优化

### 5. 可访问性

- [ ] 添加适当的 ARIA 属性
- [ ] 键盘导航支持
- [ ] 语义化 HTML 标签
- [ ] 适当的颜色对比度

## 提交前检查

创建或修改组件后，运行以下检查：

```bash
npm run format # 需要修复格式时执行
npm run check  # 完成前统一检查
```

确保：

- 代码风格符合项目规范
- 类型检查通过
- 没有 ESLint 错误或警告
- 组件功能完整且职责清晰
