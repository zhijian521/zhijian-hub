# 01 · 项目总则

- 文档状态：开发规范
- 适用范围：当前项目
- 最后核对：2026-07-19
- 维护范围：项目目标、技术边界和统一完成标准
- 关联代码：`package.json`、`tsconfig.json`、`src/`

## 目标

本项目是基于 Next.js 的前端项目，致力于建立清晰的项目结构、稳定的架构边界和可执行的开发规范。

核心目标只有三个：

- 简洁：能复用就不重写，能删掉就不保留，能一层解决就不绕两层。
- 整齐：同类东西放在同类位置，命名、分层、依赖方向稳定一致。
- 完整：功能闭环，接口有文档，组件可预览，规则有校验兜底。

## 技术栈

| 项       | 选型                         | 备注                              |
| -------- | ---------------------------- | --------------------------------- |
| 运行环境 | Node.js 24.11.1              | 以根目录 `.nvmrc` 为准            |
| 框架     | Next.js 15.5.20 App Router   | 当前只有一个前端页面入口          |
| UI       | React 19.1.0                 | 函数组件 + Hooks                  |
| 语言     | TypeScript 5.8.3             | 严格模式                          |
| 样式     | CSS Modules + CSS 变量       | 统一使用现有 token 和主题变量     |
| 状态管理 | React 本地状态 + 自定义 Hook | 当前不引入 Context 或第三方状态库 |

只有出现明确的跨层共享或跨页面状态需求时，才评估 Context 或第三方状态库；不得为未来可能出现的需求提前引入。

新增依赖前，先按这个顺序自问：

1. 现有实现能不能复用？
2. Node/浏览器原生能力能不能解决？
3. 已安装依赖能不能解决？
4. 如果都不行，再引新依赖，而且必须能说明收益。

## YAGNI 优先级

写任何东西前，按这个顺序判断，停在第一个成立的位置：

1. 这东西现在真的需要吗？
2. 项目里已经有了吗？
3. 标准库能做吗？
4. 平台原生能做吗？
5. 已装依赖能做吗？
6. 能不能用最小实现做完？

不要为了“以后可能用到”提前建接口、目录、抽象层、工厂或空文档。

## 项目边界

- 项目功能模块化组织，公共能力按统一目录规范沉淀。
- 调整实现时优先保持产品行为稳定，不为了“更优雅”随意改动既有交互、接口语义或数据规则。
- 目录结构、模块边界、规范体系和代码组织方式必须长期稳定，避免随功能推进持续漂移。

## 明确不做

- 不做过于复杂的自动化工具链。
- 不做全局的空壳抽象或未使用的架构层。
- 不做只有一个实现的接口抽象。
- 不做没有实际用处的样板代码。

- 不做没有落地校验的“建议型规范”。

## 强制校验

提交前统一运行：

```bash
npm run check
```

当前 `check` 包含：

- `npm run format:check`：代码与文档格式检查
- `npm run lint`：ESLint 代码质量检查
- `npm run typecheck`：TypeScript 类型检查
- `npm run test`：Vitest 自动化测试

需要自动修复格式时运行：

```bash
npm run format
```

“这次改动是否完成”的判断，以 `npm run check` 为准。
