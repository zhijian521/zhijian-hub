# 画板实体扩展方案

## 文档说明

- 文档状态：设计方案
- 适用版本：`zhijian-hub@0.1.0`
- 最后更新：2026-07-18
- 维护范围：尚未实施的画板实体扩展设计
- 关联代码：`src/lib/types/room-canvas.ts`、`src/lib/hooks/use-room-canvas.ts`

## 当前状态

画板当前仅包含 `Room`（房间）。房间数据由前端组件状态维护，支持创建、选择、移动、缩放、重命名、删除、搜索、聚焦和撤销/重做；当前已使用浏览器 localStorage 在刷新后恢复房间数据。

后续实体应沿着“区域 → 容器/设备 → 物品”的层级逐步扩展：

```text
Room（房间）
├── Cabinet（柜体）
│   └── Item（物品）
└── StorageDevice（储物设备）
    └── Item（物品）
```

`Item` 也可以直接归属房间，适用于未放入柜体或设备的物品。

## 推荐数据结构

在第二种画板实体真正开始开发时，再将现有单一房间状态抽为扁平、规范化的 `CanvasDocument`。不要提前加入未使用的数据模型或全局状态。

```ts
interface CanvasDocument {
    version: 1;
    rooms: Room[];
    cabinets: Cabinet[];
    storageDevices: StorageDevice[];
    items: Item[];
}

interface Cabinet {
    id: string;
    roomId: string;
    name: string;
    x: number;
    y: number;
    width: number;
    height: number;
}

interface StorageDevice {
    id: string;
    roomId: string;
    name: string;
    x: number;
    y: number;
    width: number;
    height: number;
}

interface Item {
    id: string;
    name: string;
    roomId: string;
    parentId?: string;
    parentKind?: 'cabinet' | 'storage-device';
}
```

关系通过 `roomId` 与 `parentId` 建立，不把完整子数组深层嵌进房间对象。这样单个柜体、设备或物品修改时，不需要复制整棵嵌套数据。

## 画板展示与交互

- 房间是一级可定位区域。
- 柜体和储物设备在所属房间的坐标范围内定位、选择、移动和重命名。
- 物品默认以标记或名称显示在父实体内；只有需要精确位置时，才增加独立画板坐标。
- 选中、悬停、高亮和视口聚焦应使用同一套状态语义，但不写入撤销/重做历史。
- 实体的新增、删除、重命名、位置和尺寸调整需要进入撤销/重做历史。

建议在第二种实体出现时再按职责拆分渲染组件：`RoomNode`、`CabinetNode`、`StorageDeviceNode`、`ItemMarker`。不要提前制作一个承担所有显示和编辑职责的“万能实体组件”。

## 搜索与定位

现有搜索先覆盖房间。扩展后，让搜索结果使用统一展示字段：

```ts
interface CanvasSearchResult {
    id: string;
    kind: 'room' | 'cabinet' | 'storage-device' | 'item';
    name: string;
    path: string;
    roomId: string;
}
```

`path` 用于列表展示，例如“客厅 / 北侧柜 / 工具箱”。点击结果时：

1. 先定位到目标所属房间；
2. 必要时展开或高亮父柜体、设备；
3. 高亮最终目标，并把键盘焦点交给对应画板节点。

搜索索引与类型可在第二种实体落地时抽为 `canvas-search`；在此之前继续保持搜索逻辑靠近画板功能区。

## 实施顺序

1. 新增柜体：定义 `Cabinet`，支持房间内创建、选择、移动、重命名和删除。
2. 完成柜体搜索、定位、高亮与历史记录。
3. 新增储物设备：复用已验证的“房间内可定位实体”交互，不强行共享过早抽象的渲染组件。
4. 新增物品：先实现柜体/设备中的名称、数量等信息；确认有空间规划需求后，再增加画板位置。
5. 接入后端 API 时，保留 `CanvasDocument.version` 与数据迁移；以服务端数据为主，localStorage 可作为离线缓存或草稿。

## 测试重点

- 搜索每种实体后能定位、选中并高亮正确目标。
- 删除房间时明确处理其子实体（通常应二次确认并级联删除）。
- 撤销/重做可恢复实体及其位置、尺寸、名称变化。
- 键盘导航、弹窗输入和画板浮层仍不会误触画板绘制或拖拽。
- 文档版本升级时，旧数据可迁移或给出可理解的恢复提示。

## 暂不做的内容

- 不提前引入全局 store。
- 不为只有一次使用的逻辑拆分公共层。
- 不使用深层嵌套状态保存所有实体。
- 不在未有持久化需求前引入数据库、接口或复杂迁移流程。
