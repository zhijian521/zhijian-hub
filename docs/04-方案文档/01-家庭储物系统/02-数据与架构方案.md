## 文档说明

- 文档状态：设计方案
- 适用版本：`zhijian-hub@0.1.0`
- 最后核对：2026-07-18
- 维护范围：CanvasDocument v2、实体关系、历史、迁移和模块职责
- 关联代码：`src/lib/hooks/use-room-canvas.ts`、`src/lib/types/room-canvas.ts`、`src/lib/utils/room-canvas-storage.ts`

上级方案：[家庭储物系统方案总览](./00-方案总览.md)

## 1. 数据设计原则

- 所有实体使用扁平数组保存。
- 实体关系通过 ID 建立，不把完整子数组嵌入父实体。
- 每个实体只保存直接位置，不重复保存完整路径。
- 搜索索引、位置路径和物品数量均为派生数据，不持久化。
- 存储入口必须验证父实体存在且类型符合层级规则。
- 已建立的 v2 文档只保留后续实体所需的最小字段，不提前加入标签、图片等未使用属性。

## 2. CanvasDocument v2

第一阶段已将原单一 `Room[]` 状态升级为版本化文档，后续阶段已依次启用家具、储物设备和物品。

```ts
interface CanvasDocumentV2 {
    version: 2;
    counters: {
        room: number;
        furniture: number;
        storageDevice: number;
        item: number;
    };
    rooms: Room[];
    furniture: Furniture[];
    storageDevices: StorageDevice[];
    items: Item[];
}
```

计数器只用于生成稳定默认名称，不进入撤销/重做历史。

## 3. 实体模型

### 3.1 房间

继续使用当前房间模型和全局画布坐标：

```ts
interface Room {
    id: string;
    name: string;
    x: number;
    y: number;
    width: number;
    height: number;
}
```

### 3.2 家具

```ts
interface Furniture {
    id: string;
    roomId: string;
    name: string;
    x: number;
    y: number;
    width: number;
    height: number;
}
```

家具坐标相对于所属房间左上角，不保存全局画布坐标。

### 3.3 储物设备

```ts
interface StorageDeviceBase {
    id: string;
    name: string;
}

type StorageDevice =
    | (StorageDeviceBase & {
          location: { kind: 'room'; roomId: string };
          rect: { x: number; y: number; width: number; height: number };
      })
    | (StorageDeviceBase & {
          location: { kind: 'furniture'; furnitureId: string };
      });
```

只有直接放在房间中的储物设备需要画布矩形。家具内部设备的初始实现只维护父子关系。

### 3.4 物品

```ts
type ItemLocation =
    | { kind: 'room'; roomId: string }
    | { kind: 'furniture'; furnitureId: string }
    | { kind: 'storage-device'; storageDeviceId: string };

interface Item {
    id: string;
    name: string;
    quantity: number;
    location: ItemLocation;
}
```

物品名称在初始实现中允许重复，搜索结果通过完整位置路径区分。

## 4. 关系与完整性

### 4.1 父级限制

- `Furniture.roomId` 必须指向存在的房间。
- 房间级储物设备必须指向存在的房间，并包含合法矩形。
- 家具内储物设备必须指向存在的家具。
- 物品位置必须指向存在且类型匹配的房间、家具或储物设备。

### 4.2 路径解析

通过实体 ID 查找表实时计算路径：

```text
Item → StorageDevice → Furniture → Room
Item → Furniture → Room
Item → Room
```

最大层级深度固定，不需要建立复杂树缓存。实体数量处于家庭使用规模时，前端内存解析足够使用。

### 4.3 位置移动

- 家具更换 `roomId` 后，其后代路径自动变化。
- 储物设备切换位置联合类型时，增加或移除 `rect`。
- 物品只替换 `location`，不重写名称路径。

## 5. 搜索数据

```ts
type SearchEntityKind = 'room' | 'furniture' | 'storage-device' | 'item';

interface CanvasSearchResult {
    id: string;
    kind: SearchEntityKind;
    name: string;
    path: string[];
    roomId: string;
}
```

搜索结果由当前文档实时生成，不写入 `CanvasDocumentV2`。当前四类实体均按名称进行忽略大小写的包含匹配，家具、储物设备和物品路径由父级关系实时解析；只有出现明确需求时再增加标签或别名。

## 6. 历史记录

第一阶段已将历史升级为完整画板内容快照，包含：

- 房间。
- 家具。
- 储物设备。
- 物品。

以下状态不进入历史：

- 实体默认命名计数器。
- 当前选择和高亮。
- 搜索词和面板开关。
- 工具切换。
- 画布平移和缩放。
- 定位动画状态。

级联删除、位置转移等复合操作必须作为一次提交进入历史。当前保留最多 50 条完整快照，只有实际数据规模产生明显问题时才评估命令模式或增量补丁。

## 7. 本地数据迁移

历史存储文档 `version: 1` 只包含房间和房间序号，当前保存格式已升级为 `version: 2`。兼容迁移路径：

```text
version 1
└── rooms

version 2
├── rooms              # 保留原数据
├── furniture: []
├── storageDevices: []
└── items: []
```

迁移要求：

- 保留现有房间的 ID、名称、坐标和尺寸。
- 把现有房间序号迁移到 `counters.room`。
- 新实体计数器从 0 开始。
- 迁移失败时不得覆盖原始 localStorage 数据。
- 加载器同时支持读取 version 1 和 version 2，保存时统一写入 version 2。
- 正常迁移、异常结构和不兼容版本必须有自动化测试。

下一阶段增加 JSON 导入和导出，降低浏览器数据被清理后的丢失风险。

## 8. 架构职责调整

第一阶段已抽出 `useCanvasDocument` 管理领域状态、历史和持久化；`useRoomCanvas` 继续负责房间指针交互、选择和视口。

```text
画板协调层
├── 文档状态       # 实体数据、增删改、历史和持久化
├── 视口状态       # 平移、缩放、定位和高亮
└── 空间交互       # 绘制、移动和调整大小
```

后续拆分顺序：

1. `use-canvas-document.ts` 已管理版本化文档、完整内容历史和持久化。
2. 当前 `use-room-canvas.ts` 继续管理房间指针交互和视口。
3. 房间、家具与房间级储物设备继续共享同一套指针交互监听；只有复杂度继续增长时再提取空间行为。
4. `RoomNode`、`FurnitureNode` 和 `StorageDeviceNode` 已在 `canvas-nodes.tsx` 中按具体实体拆出。
5. `CanvasActions` 负责搜索和面板壳，`CanvasDetails` 负责容器详情，`CanvasItems` 负责物品列表与编辑表单。
6. 不制作承担所有实体显示和交互的万能组件。

当前不引入 Context、Zustand 或其他全局状态库。只有出现明确跨页面或跨层级共享压力时再评估。

## 9. 性能策略

- 家庭规模数据先使用数组和 ID 查找表，不引入数据库索引结构。
- 指针移动继续使用 `requestAnimationFrame` 合并更新。
- 搜索索引只在出现实际性能问题时缓存。
- 历史快照只在完整操作结束时记录。
- 物品不作为画布节点渲染，避免大量 DOM 元素。

## 10. 主要技术风险

| 风险                            | 控制方式                               |
| ------------------------------- | -------------------------------------- |
| 深层嵌套状态难以维护            | 使用扁平数组和 ID 关系                 |
| 重复保存位置路径导致过期        | 路径实时派生，不持久化                 |
| localStorage 被清理导致数据丢失 | 物品阶段完成后增加 JSON 导入导出       |
| 画板 Hook 继续膨胀              | 文档职责已抽离，新增实体时保持现有边界 |
| 全量快照随数据增长占用内存      | 先监测真实规模，再决定是否改为增量历史 |
| 过早抽象拖慢开发                | 每种实体落地时再拆对应组件和共享逻辑   |
