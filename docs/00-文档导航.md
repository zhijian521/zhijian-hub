## 3. 推荐阅读顺序

1. 根目录 `README.md`：了解项目定位、运行方式和质量命令。
2. 《需求文档》：确认当前产品范围和验收边界。
3. 《功能文档》：了解页面操作和行为细节。
4. 《架构设计文档》：了解代码结构和核心数据流。
5. 《开发规范导航》：开始开发前读取对应专项规范。
6. 方案文档：只在实现相关规划时阅读。

## 4. 命名与编号规则

- 顶层分类目录使用两位数字编号，表达稳定的阅读顺序。
- 分类内部独立编号，从 `00` 或 `01` 开始；`00` 只用于导航索引。
- 文档文件名使用中文，保留必要的通用技术名称，例如 Git。
- 文件名不包含日期和版本号，适用版本或范围放在文档头部。
- 新增文档时先确定类别，再选择该类别下的下一个编号。
- 不因插入单个文件批量重排既有编号，避免大量无意义链接变更。

## 5. 文档头部格式

每份正式文档在一级标题后维护以下信息：

```markdown
- 文档状态：当前实现基线 / 当前功能说明 / 开发规范 / 设计方案 / 已废弃
- 适用版本或范围：`zhijian-hub@x.y.z` / 当前项目
- 最后核对：YYYY-MM-DD
- 维护范围：本文档负责描述的内容
- 关联代码：相关目录、配置或核心文件
```

产品和技术基线文档保留适用版本；长期生效的开发规范使用“适用范围：当前项目”。只有重新核对过正文与代码后才更新“最后核对”日期。

## 6. 文档拆分规则

- 少于 300 行且主题单一时，优先保留一个文件。
- 超过 400～500 行时，检查是否存在可独立维护的主题，再决定拆分。
- 文件包含三个以上相互独立、更新频率不同的主题时，可以提前拆分。
- 拆分后的分类目录保留一份总览文档，子文档按阅读顺序编号。
- 不为尚未出现的复杂度提前创建空文件或空目录。

当前需求、功能和架构文档仍可独立维护，暂不继续拆分。

## 7. 同步维护规则

代码和文档必须在同一次提交中保持一致：

| 代码变化                               | 必须检查或更新的文档                 |
| -------------------------------------- | ------------------------------------ |
| 新增、删除或改变用户功能               | 需求文档、功能文档                   |
| 修改快捷键、按钮、默认值或交互行为     | 需求文档、功能文档                   |
| 修改目录、模块职责、数据模型或状态流转 | 架构设计文档                         |
| 修改存储格式、历史策略或坐标规则       | 架构设计文档；影响用户时同步产品文档 |
| 修改开发约束、工具链或质量门禁         | 对应开发规范、规范导航和 README      |
| 实施现有设计方案                       | 对应方案文档、产品文档和技术文档     |

只调整内部实现且不改变用户行为和架构边界时，无需机械修改所有文档，但必须确认现有描述仍然准确。

## 8. 文档状态管理

- 当前实现基线：描述已经落地并可验证的系统事实。
- 当前功能说明：描述用户现在可以使用的功能。
- 开发规范：约束后续代码和协作方式。
- 设计方案：记录尚未实施的设计，不混入当前功能基线。
- 已废弃：内容不再适用；存在保留价值时再移动到 `99-归档/`。

没有历史价值的过期文档直接删除，不同时保留多份内容相近的有效文档。
